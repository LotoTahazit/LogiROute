## Назначение

Этот документ является единым источником контекста для AI-ассистентов (Codex, Cursor, ChatGPT и других), работающих над проектом **LogiRoute**.

Перед выполнением любых изменений AI должен сначала прочитать этот документ.

# О проекте

**Название:** LogiRoute

**Тип продукта:**
SaaS-платформа для логистики и дистрибуции.

**Целевая аудитория:**

* дистрибьюторы
* оптовые компании
* службы доставки
* компании со своим складом
* SMB Израиля

---

# Бизнес-цели проекта

LogiRoute создаётся не как демонстрация технологий, а как коммерческий SaaS-продукт.

Главная цель — заменить для малого и среднего дистрибьютора несколько отдельных систем одной платформой.

Типичный клиент сегодня использует:

* Excel
* WhatsApp
* Google Maps / Waze
* отдельную бухгалтерскую систему
* отдельный учёт склада

LogiRoute объединяет эти процессы.

---

# Целевая компания

Размер компании:

* 5–30 водителей

* 200–5000 SKU

* Израиль

Оптимизация под enterprise не является приоритетом до успешного пилота.

# Архитектурные принципы

* Не дублировать существующие экраны.
* Не дублировать бизнес-логику.
* Использовать существующие сервисы.
* Новая функциональность должна переиспользовать существующую архитектуру.
* Любое изменение должно быть обратно совместимым.

## Архитектурные правила

Запрещается:

* дублировать бизнес-логику;
* создавать второй сервис для уже существующей функции;
* выполнять Firestore full read без необходимости;
* создавать новые модели без анализа существующих;
* обходить ModuleGuard или BillingGuard.

Предпочитать переиспользование существующих компонентов.

---

# Принципы UX

Любой экран должен отвечать на три вопроса:

1. Что происходит?
2. Что мне делать дальше?
3. Что случится после нажатия?

Не использовать технические сообщения для пользователя.

Все ошибки должны объясняться простым языком.

---

# Работа с Firestore

Использовать:

* пагинацию;
* `limit()`;
* курсоры;
* диапазоны дат;
* server-side фильтрацию.

Избегать:

* full collection reads;
* бесконтрольных realtime stream;
* лишних writes.

Все новые запросы должны оцениваться с точки зрения стоимости Firestore.

---

# Работа с Cloud Functions

Cloud Functions должны быть:

* идемпотентными;
* безопасными для повторного запуска;
* не зависеть от порядка выполнения;
* корректно логировать ошибки;
* использовать `correlationId`, если он доступен.

---

# Безопасность

При любой неопределённости выбирать безопасное поведение.

Например:

* неизвестный billing → блокировать доступ;
* неизвестная роль → не давать доступ;
* неизвестная компания → отказ;
* отсутствие разрешения → отказ.

**Fail Open запрещён.** Предпочтителен **Fail Closed**.

---

# Логирование

Критические операции должны содержать:

* `companyId`
* `userId`
* `correlationId`
* `timestamp`
* `operation`
* результат

Все ошибки должны иметь понятное сообщение для пользователя и подробную запись для поддержки.

# Текущий этап развития

Проект находится на стадии подготовки к первому коммерческому пилоту.

Главная задача — не добавление функций, а:

* стабильность;
* простота внедрения;
* минимизация рисков;
* успешный запуск первой компании.

Любые крупные новые функции (AI, BI, расширенная CRM, Enterprise-возможности) должны быть отложены до завершения пилота и анализа реального использования.

# Appendix A — Architecture Decision Records (ADR)

## Назначение

Этот раздел фиксирует ключевые архитектурные решения проекта.

AI не должен предлагать изменения, противоречащие этим решениям, без явного обоснования.

Каждое решение считается осознанным компромиссом.

---

## ADR-001 — Единая платформа вместо отдельных систем

### Решение

LogiRoute объединяет:

* логистику;
* склад;
* бухгалтерию;
* GPS;
* документы;
* отчёты;
* биллинг.

### Причина

Основная ценность продукта — единый источник данных.

Любое решение, разделяющее эти процессы обратно на независимые подсистемы, считается нежелательным.

---

## ADR-002 — Firestore как основная база данных

### Решение

Firestore является основной operational database.

### Причина

Проект ориентирован на:

* быстрый запуск;
* минимальное администрирование;
* масштабирование SaaS;
* тесную интеграцию с Firebase.

Предложения заменить Firestore допускаются только при наличии объективных ограничений.

---

## ADR-003 — Fail Closed

### Решение

Если невозможно проверить:

* права;
* подписку;
* компанию;
* разрешения;

доступ запрещается.

### Причина

Безопасность важнее удобства.

---

## ADR-004 — BillingGuard обязателен

BillingGuard нельзя обходить.

Если статус подписки неизвестен — показывается экран ошибки проверки.

---

## ADR-005 — Company Setup Wizard

Wizard не содержит собственной бизнес-логики.

Он использует существующие экраны.

Любое изменение Wizard должно сохранять этот принцип.

---

## ADR-006 — Demo Mode

Demo Company полностью изолирована.

Все demo-данные имеют:

* `demoCompany=true`
* `isDemo=true`

Reset Demo никогда не должен работать для реальной компании.

---

## ADR-007 — Correlation ID

Все критические операции используют единый `correlationId`.

Если `requestId` уже существует — он становится `correlationId`.

Создание двух разных идентификаторов запрещено.

---

## ADR-008 — Pagination

Полные чтения коллекций запрещены для пользовательских экранов.

Допускаются только:

* экспорт;
* обслуживание;
* административные операции.

Все пользовательские списки должны использовать:

* `limit()`;
* pagination;
* server-side filtering.

---

## ADR-009 — Viewer

Роль `viewer` временно отключена.

**Причина:** не существует полноценного read-only интерфейса.

До появления такого интерфейса назначение viewer запрещено.

---

## ADR-010 — Restore Drill

Отметка «Restore Success» невозможна без подтверждения.

Каждый успешный restore должен иметь evidence.

---

## ADR-011 — Метрики

Owner Dashboard никогда не должен показывать фиктивные KPI.

Если метрики отсутствуют — показывается сообщение: «Метрики ещё не рассчитаны.»

---

## ADR-012 — Demo вместо пустых экранов

Система должна демонстрироваться на полноценной Demo Company.

Пустые экраны во время презентации считаются дефектом продукта.

---

## ADR-013 — Минимизация новых сервисов

Перед созданием нового сервиса AI обязан проверить возможность использования существующих.

Дублирование сервисов считается архитектурной ошибкой.

---

## ADR-014 — Product First

На текущем этапе проекта приоритеты следующие:

1. Стабильность.
2. Внедрение.
3. Продажи.
4. Поддержка.
5. Масштабирование.
6. Новые функции.

AI не должен предлагать крупные новые модули, пока не завершён первый коммерческий пилот.

---

## ADR-015 — Решения должны уменьшать риск потери первого клиента

Главный критерий оценки любого изменения:

> Уменьшает ли это вероятность потерять первого коммерческого клиента?

Если ответ отрицательный — изменение переносится в backlog.

---

## ADR-016 — Архитектурная философия

LogiRoute проектируется как долгоживущий коммерческий продукт.

Поэтому предпочтение всегда отдаётся:

* простым решениям;
* читаемому коду;
* минимальному количеству компонентов;
* предсказуемому поведению;
* удобству сопровождения.

Красивое инженерное решение никогда не должно иметь приоритет над надёжностью и понятностью системы.

# Appendix B — Lessons Learned

## Назначение

Этот раздел содержит практические уроки, полученные в ходе разработки LogiRoute.

Это не список ошибок. Это список решений, которые уже были проверены практикой.

Перед предложением изменений AI обязан проверить, не противоречат ли они этим выводам.

---

## LL-001 — Fail Open почти всегда приводит к проблемам

### Было

BillingGuard пропускал пользователя при timeout или неизвестном статусе.

### Стало

Любая невозможность проверить подписку приводит к временному отказу в доступе (Fail Closed).

### Правило

Неизвестность никогда не должна трактоваться как разрешение.

---

## LL-002 — Пользователь не должен видеть ложный успех

### Было

Stripe Checkout сообщал об успехе даже тогда, когда страница оплаты не открылась.

### Стало

Успех отображается только после успешного открытия Checkout.

### Правило

Никогда не показывать Success без фактического результата.

---

## LL-003 — Зелёная иконка не должна врать

### Было

GPS отображался как активный при выключенной геолокации.

### Стало

UI показывает реальное состояние GPS.

### Правило

Статус интерфейса обязан соответствовать фактическому состоянию системы.

---

## LL-004 — Пустые кнопки хуже отсутствующих

### Было

Support и Upgrade ничего не делали.

### Стало

Все действия либо работают, либо объясняют пользователю, почему действие недоступно.

### Правило

В интерфейсе не должно быть элементов без поведения.

---

## LL-005 — Полные чтения Firestore плохо масштабируются

### Было

Reports, Analytics, Clients, Inventory читали всю коллекцию.

### Стало

Все пользовательские запросы ограничены диапазонами, лимитами и пагинацией.

### Правило

Full Read допустим только для административных операций и экспорта.

---

## LL-006 — Dashboard без данных выглядит как неисправность

### Было

Owner Dashboard показывал пустые KPI.

### Стало

Добавлен расчёт метрик и понятное сообщение при их отсутствии.

### Правило

Лучше честно показать отсутствие данных, чем отображать фиктивные значения.

---

## LL-007 — Demo должна выглядеть как реальная компания

### Было

Пустые экраны производили плохое впечатление.

### Стало

Demo Company содержит реалистичные данные.

### Правило

Демонстрация должна показывать возможности продукта, а не процесс его настройки.

---

## LL-008 — Wizard должен объединять существующие функции

### Было

Возникало желание создать отдельную бизнес-логику настройки.

### Стало

Company Setup Wizard использует существующие экраны.

### Правило

Wizard — это сценарий, а не новая подсистема.

---

## LL-009 — Correlation ID значительно сокращает время поддержки

### Было

Поиск причины ошибки занимал много времени.

### Стало

Критические операции связаны единым `correlationId`.

### Правило

Любая операция, влияющая на бизнес-процесс, должна быть трассируемой.

---

## LL-010 — Restore считается успешным только после проверки

### Было

Достаточно было поставить галочку.

### Стало

Успешный Restore требует подтверждающих данных (evidence).

### Правило

Disaster Recovery должен регулярно проверяться, а не только документироваться.

---

## LL-011 — Не все технические улучшения одинаково ценны

Во время подготовки продукта был изменён приоритет разработки.

**Было:**

* новые функции;
* улучшение архитектуры;
* оптимизация.

**Стало:**

* уменьшение риска потери первого клиента;
* простота внедрения;
* стабильность;
* удобство поддержки.

### Правило

Главный критерий выбора задачи:

> Поможет ли это успешному запуску и удержанию первого коммерческого клиента?

Если нет — задача переносится в backlog.

---

## LL-012 — Пилот важнее предположений

Никакие архитектурные обсуждения не заменяют реальное использование системы.

После начала коммерческого пилота решения должны приниматься на основе:

* поведения пользователей;
* времени выполнения операций;
* обращений в поддержку;
* фактических затрат Firebase;
* обратной связи клиентов.

### Правило

Фактические данные всегда имеют приоритет над предположениями.

---

## Заключение (Appendix B)

LogiRoute проектируется не как демонстрационный проект и не как учебный пример.

Каждое архитектурное решение должно повышать:

* надёжность;
* сопровождаемость;
* простоту внедрения;
* удобство использования;
* коммерческую успешность продукта.

Если новое предложение ухудшает хотя бы один из этих пунктов, оно должно быть серьёзно обосновано перед внедрением.

# Appendix C — Non-Negotiable Rules

## Назначение

Ниже перечислены правила, которые считаются обязательными для всех изменений в проекте LogiRoute.

Если предлагаемое изменение нарушает хотя бы одно из этих правил, AI обязан:

1. предупредить об этом;
2. объяснить последствия;
3. предложить альтернативное решение.

Эти правила имеют приоритет над любыми рекомендациями по «красивому» или «современному» коду.

---

## NN-001

Никогда не дублировать существующую бизнес-логику.

Перед созданием нового сервиса или метода необходимо проверить, существует ли уже аналогичная функциональность.

---

## NN-002

Никогда не делать Full Firestore Read на пользовательских экранах.

Разрешены только:

* экспорт;
* административные операции;
* обслуживание;
* миграции.

Во всех остальных случаях использовать:

* pagination;
* `limit()`;
* cursor;
* server-side filtering.

---

## NN-003

Fail Open запрещён.

Если невозможно проверить:

* права;
* подписку;
* роль;
* компанию;
* разрешения;

доступ запрещается.

---

## NN-004

Пользователь никогда не должен видеть ложный Success.

Любой Success должен соответствовать фактически выполненной операции.

---

## NN-005

Любая ошибка должна:

* логироваться;
* иметь `correlationId` (если применимо);
* иметь понятное сообщение пользователю.

Пустые `catch {}` запрещены.

---

## NN-006

Все критические операции должны быть трассируемыми.

Минимальный набор:

* `companyId`;
* `userId`;
* `correlationId`;
* `operation`;
* `timestamp`.

---

## NN-007

Не создавать новый экран, если можно использовать существующий.

Wizard, Demo и onboarding должны быть навигацией поверх существующих компонентов.

---

## NN-008

Любое изменение должно быть обратно совместимым.

Не ломать существующие компании, документы, маршруты или данные без явной миграции.

---

## NN-009

Не использовать «магические» значения.

Все настройки должны быть вынесены в:

* AppConfig;
* Company Settings;
* Feature Flags;
* Billing Plan.

---

## NN-010

Demo никогда не должна иметь доступ к данным реальных компаний.

Все demo-объекты должны иметь явную маркировку.

---

## NN-011

Любое удаление данных должно быть безопасным.

Перед удалением необходимо убедиться, что операция не затронет другую компанию (tenant).

---

## NN-012

Никогда не обходить:

* BillingGuard;
* ModuleGuard;
* Firestore Rules.

Временные обходы («только на время разработки») запрещены.

---

## NN-013

Cloud Functions должны быть идемпотентными.

Повторный запуск функции не должен приводить к повреждению данных или дублированию документов.

---

## NN-014

Не увеличивать стоимость Firestore без необходимости.

Каждый новый запрос должен оцениваться с точки зрения:

* количества Reads;
* количества Writes;
* количества Streams;
* потенциальной стоимости при росте клиентов.

---

## NN-015

Не добавлять новые зависимости без веской причины.

Сначала использовать возможности существующего стека.

---

## NN-016

Не создавать «умные» решения там, где достаточно простого.

Простое, понятное и сопровождаемое решение имеет приоритет.

---

## NN-017

Никогда не показывать пользователю технические ошибки.

Сообщение должно объяснять:

* что произошло;
* что можно сделать дальше.

Подробности остаются в логах.

---

## NN-018

Все пользовательские действия должны быть предсказуемыми.

Кнопка не может:

* ничего не делать;
* неожиданно менять данные;
* скрыто выполнять длительные операции без индикации.

---

## NN-019

Не добавлять новые функции до завершения текущего этапа проекта.

Текущий этап определяется разделом «Текущий этап развития» настоящего документа.

---

## NN-020

Приоритет изменений определяется не сложностью задачи, а влиянием на бизнес.

Перед началом работы AI обязан ответить:

* уменьшает ли изменение риск потери первого клиента;
* упрощает ли внедрение;
* уменьшает ли нагрузку на поддержку;
* повышает ли надёжность системы.

Если на все вопросы ответ отрицательный, задача должна быть перенесена в backlog.

---

## NN-021

Любое изменение должно сопровождаться проверкой.

Минимум:

* список изменённых файлов;
* описание причины изменения;
* сценарии ручной проверки;
* при наличии — unit/integration tests.

Изменение без плана проверки считается незавершённым.

---

## NN-022

Документация является частью продукта.

Если изменение влияет на:

* архитектуру;
* роли;
* бизнес-процесс;
* развёртывание;
* внедрение;
* безопасность;

необходимо обновить соответствующую документацию.

---

## NN-023

Архитектурные решения фиксируются, а не теряются.

Любое изменение, затрагивающее ранее принятые ADR, должно:

* ссылаться на соответствующий ADR;
* объяснять причину отклонения;
* обновлять документацию при принятии нового решения.

---

## NN-024

Данные реального клиента всегда важнее новых функций.

При конфликте между:

* разработкой новой возможности;
* сохранностью данных;
* стабильностью работы;
* восстановлением после сбоя;

приоритет имеют данные и стабильность.

---

## NN-025 — Главное правило проекта

Каждое изменение должно делать LogiRoute:

* проще для пользователя;
* надёжнее в эксплуатации;
* дешевле в сопровождении;
* легче во внедрении;
* сильнее как коммерческий продукт.

Если изменение не улучшает хотя бы один из этих пунктов, оно должно быть пересмотрено или отложено.

---

# Appendix D — Definition of Done (DoD)

## Назначение

Этот раздел определяет единые критерии завершения любой задачи в проекте LogiRoute.

Задача не считается выполненной, пока не выполнены все обязательные пункты.

---

## DoD-001 — Код компилируется

Изменения не должны ломать сборку проекта.

Обязательно:

* Flutter Analyze без новых ошибок;
* сборка Web проходит;
* сборка Android проходит (если изменение затрагивает Android).

---

## DoD-002 — Архитектура не нарушена

Изменение:

* не дублирует существующие сервисы;
* не нарушает ADR;
* не создаёт новый способ решения уже существующей задачи.

---

## DoD-003 — Firestore

Если изменение затрагивает Firestore, обязательно проверить:

* rules;
* indexes;
* стоимость запросов;
* cross-tenant безопасность.

Нельзя добавлять full collection read без письменного обоснования.

---

## DoD-004 — Безопасность

Проверить:

* BillingGuard;
* ModuleGuard;
* роли;
* права;
* Firestore Rules.

Нельзя создавать новый путь обхода существующих ограничений.

---

## DoD-005 — UX

Пользователь должен понимать:

* что происходит;
* что делать дальше;
* почему произошла ошибка.

Не допускаются:

* пустые кнопки;
* silent fail;
* ложный Success;
* непонятные технические сообщения.

---

## DoD-006 — Логирование

Если операция критична, она должна содержать:

* `companyId`;
* `userId`;
* `correlationId`;
* `timestamp`;
* `operation`.

Ошибки должны логироваться.

---

## DoD-007 — Документация

Если изменение влияет на:

* архитектуру;
* роли;
* бизнес-процесс;
* внедрение;
* безопасность;
* API;

необходимо обновить соответствующий документ.

---

## DoD-008 — Отчёт AI

После выполнения задачи AI обязан предоставить отчёт. Минимум:

* **Изменённые файлы**
* **Причина изменений**
* **Что исправлено**
* **Что осталось**
* **Что проверить вручную**

---

## DoD-009 — Тестирование

Если в проекте существуют тесты, необходимо:

* обновить существующие;
* добавить новые при необходимости.

Если тестов нет, AI обязан предоставить сценарии ручной проверки.

---

## DoD-010 — Производительность

Изменение не должно:

* увеличивать Firestore Reads без причины;
* увеличивать количество Streams;
* ухудшать время загрузки экранов;
* создавать дополнительные циклы обновления.

---

## DoD-011 — Обратная совместимость

Изменение не должно ломать:

* существующие компании;
* документы;
* маршруты;
* настройки;
* данные пользователей.

Если миграция необходима, она должна быть описана.

---

## DoD-012 — Коммерческий критерий

Перед завершением задачи AI обязан ответить: это изменение

* уменьшает риск потери клиента?
* упрощает внедрение?
* уменьшает обращения в поддержку?
* повышает стабильность?

Если ответ «нет» на все вопросы, необходимо объяснить, зачем задача выполнялась.

---

## DoD-013 — Готовность к пилоту

Если изменение относится к этапу подготовки пилота, оно должно пройти:

* техническую проверку;
* UX-проверку;
* сценарий первого использования;
* проверку документации.

---

## DoD-014 — Готовность к релизу

Изменение считается готовым к релизу только если:

* отсутствуют известные критические дефекты;
* обновлена документация;
* проведена ручная проверка;
* не осталось блокеров уровня 🔴.

---

## DoD-015 — Финальное правило

«Код написан» не означает «задача завершена».

Задача считается завершённой только тогда, когда:

* код работает;
* пользователь понимает результат;
* поддержка сможет диагностировать проблему;
* документация обновлена;
* изменение безопасно для существующих клиентов;
* продукт стал лучше.

---

# Appendix E — Coding Standards

## Назначение

Этот раздел определяет единые стандарты разработки проекта **LogiRoute**.

Цель — обеспечить единообразие кода, облегчить сопровождение и минимизировать вероятность архитектурных ошибок.

Эти правила обязательны для всех новых компонентов проекта.

---

## CS-001 — Общий принцип

Код должен быть:

* простым;
* читаемым;
* предсказуемым;
* легко сопровождаемым.

Предпочитать понятный код «умному».

---

## CS-002 — Один сервис — одна ответственность

Каждый сервис отвечает только за одну область.

Например:

* `RouteService` — маршруты
* `InvoiceService` — документы
* `InventoryService` — склад
* `ClientService` — клиенты
* `BillingService` — подписка

Не смешивать разные обязанности.

---

## CS-003 — Не создавать дубликаты

Перед созданием нового сервиса проверить:

* существует ли похожий сервис;
* существует ли похожий метод.

Если существует — расширять его, а не создавать новый.

---

## CS-004 — Firestore

Запрещается `collection.get()` без необходимости.

Предпочитать:

* `where()`
* `limit()`
* `orderBy()`
* `startAfter()`
* `endBefore()`

Любой новый запрос должен быть оценён с точки зрения стоимости Reads.

---

## CS-005 — Streams

Использовать Stream только там, где действительно нужен realtime.

Не создавать несколько `StreamBuilder`, если можно использовать один.

Избегать глубоких вложенных `StreamBuilder`.

---

## CS-006 — Cloud Functions

Каждая функция должна:

* быть идемпотентной;
* корректно обрабатывать повторный вызов;
* использовать `correlationId`;
* логировать ошибки.

Никаких «тихих» исключений.

---

## CS-007 — Ошибки

Запрещается:

```dart
catch (_) {}
```

или

```dart
catch (e) {
}
```

Каждая ошибка должна:

* логироваться;
* иметь user-friendly сообщение;
* иметь техническое описание для поддержки.

---

## CS-008 — Correlation Context

Все критические операции должны использовать `CorrelationContext` или существующий `requestId`.

Нельзя создавать второй идентификатор, если первый уже существует.

---

## CS-009 — UI

Каждый экран обязан иметь:

* Loading;
* Empty;
* Error;
* Success.

Пользователь никогда не должен видеть:

* бесконечный spinner;
* пустой экран;
* stack trace;
* техническое исключение.

---

## CS-010 — Навигация

Не создавать дублирующие экраны.

Company Setup Wizard, Demo, Onboarding должны использовать существующие страницы.

---

## CS-011 — Guards

Все защищённые экраны должны проходить через:

* BillingGuard;
* ModuleGuard;
* Permission Check.

Обход запрещён.

---

## CS-012 — Роли

Любая новая роль должна иметь:

* UI;
* Firestore Rules;
* Permissions;
* Routing;
* документацию.

Запрещается добавлять роль, которая не имеет рабочего интерфейса.

---

## CS-013 — Документы

Каждый новый тип документа обязан иметь:

* модель;
* сервис;
* PDF;
* аудит;
* экспорт;
* бухгалтерскую интеграцию (если применимо).

---

## CS-014 — Конфигурация

Запрещается использовать «магические» значения.

Настройки должны храниться в:

* AppConfig;
* Company Settings;
* Billing Plan;
* Feature Flags.

---

## CS-015 — Demo

Все demo-объекты обязаны иметь:

* `demoCompany == true`
* `isDemo == true`

Reset Demo не должен работать без этих признаков.

---

## CS-016 — Документация

Любая архитектурная задача должна обновлять:

* `docs/project-structure.md`;
* `AI_MASTER_CONTEXT.md` (если изменяется архитектурное правило);
* ADR (если изменяется принятое решение).

---

## CS-017 — Pull Request Checklist

Перед завершением задачи разработчик или AI должен ответить:

* Код компилируется?
* Нет новых warning?
* Нет Full Reads?
* Нет новых Security Gaps?
* Документация обновлена?
* Добавлены тесты или сценарии проверки?
* Нет регрессий?

Если хотя бы один ответ «нет», задача не считается завершённой.

---

## CS-018 — Работа AI

Перед любым изменением AI обязан:

1. Прочитать `AI_MASTER_CONTEXT.md`.
2. Проверить ADR (Appendix A).
3. Проверить Non-Negotiable Rules (Appendix C).
4. Проверить Lessons Learned (Appendix B).
5. Проверить Definition of Done (Appendix D).
6. Следовать Coding Standards (Appendix E).

Только после этого предлагать изменения.

---

## CS-019 — Приоритет разработки

Порядок приоритетов:

1. Безопасность.
2. Надёжность.
3. Простота.
4. Производительность.
5. UX.
6. Масштабируемость.
7. Новые функции.

Нельзя нарушать этот порядок без серьёзного обоснования.

---

## CS-020 — Финальная философия проекта

LogiRoute — это коммерческий продукт, а не техническая демонстрация.

Каждое изменение должно делать систему:

* легче внедряемой;
* проще для пользователей;
* дешевле в сопровождении;
* стабильнее;
* понятнее;
* безопаснее.

Если изменение делает код «красивее», но не делает продукт лучше, оно должно быть пересмотрено.
