# מדיניות תגובה לאירועי אבטחה — Incident Response Policy

## 1. הגדרות

| מונח | הגדרה |
|------|--------|
| אירוע אבטחה | כל מצב שמסכן שלמות, סודיות או זמינות הנתונים |
| חדירה | גישה לא מורשית לנתונים או למערכת |
| דליפת מידע | חשיפת נתונים לגורם לא מורשה |
| תקלת שלמות | שינוי לא מורשה במסמך חשבונאי |

## 2. סיווג אירועים

| רמה | תיאור | דוגמאות |
|------|--------|---------|
| קריטי | חדירה / אובדן נתונים / שינוי מסמכים | bypass Rules, DB corruption |
| גבוה | ניסיון חדירה / anomaly ביומנים | failed auth spike, unusual access |
| בינוני | תקלת אבטחה ללא חשיפה | misconfigured Rule (caught) |
| נמוך | חשד ללא אישור | suspicious login pattern |

## 3. תהליך תגובה

### שלב 1: זיהוי (Detection)
- ניטור Cloud Audit Logs
- ניטור Access Log anomalies
- ניטור Audit Log — שינויים חריגים
- Firebase Crashlytics alerts
- דיווח ממשתמשים

### שלב 2: בלימה (Containment)
- השבתת חשבונות חשודים
- חסימת IP (אם רלוונטי)
- הפעלת maintenance mode (אם נדרש)
- שמירת ראיות (logs, snapshots)

### שלב 3: הערכה (Assessment)
- היקף: אילו נתונים/חברות נפגעו
- סוג: חדירה / דליפה / שינוי
- משך: מתי התחיל ומתי נעצר
- שלמות: `verifyChain()` + `verifySequentialIntegrity()`

### שלב 4: תיקון (Remediation)
- תיקון הפגיעות
- שחזור נתונים (אם נדרש)
- עדכון Rules / Auth
- אימות שלמות לאחר תיקון

### שלב 5: הודעה (Notification)
- הודעה ללקוחות נפגעים (תוך 72 שעות — GDPR)
- הודעה לרשות להגנת הפרטיות (אם נדרש)
- הודעה לרואה חשבון (אם נפגעו מסמכים חשבונאיים)

### שלב 6: למידה (Post-mortem)
- דוח אירוע מפורט
- שורש הבעיה (root cause)
- צעדי מניעה
- עדכון מדיניות

## 4. אנשי קשר

| תפקיד | אחריות | זמן תגובה |
|--------|---------|-----------|
| מנהל מערכת | הפעלת תוכנית, תקשורת | 1 שעה |
| מפתח ראשי | חקירה טכנית, תיקון | 2 שעות |
| יועץ משפטי | הודעות, GDPR | 24 שעות |

## 5. כלי ניטור

| כלי | מטרה |
|------|-------|
| Cloud Audit Logs | גישה ישירה ל-Firestore |
| Access Log (app) | פעולות משתמשים |
| Audit Log (app) | שינויים במסמכים |
| Firebase Crashlytics | שגיאות אפליקציה |
| Integrity Chain | אימות שלמות |

## 6. תרגול

- סימולציית אירוע: פעם בשנה
- סקירת מדיניות: פעם בשנה
- עדכון אנשי קשר: פעם ברבעון
