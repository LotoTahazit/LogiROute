# אי-שינוי ושלמות מסמכים (Immutability & Integrity)

> **מסמך זה נדרש להגשה לרישום תוכנה לניהול ספרים (מס הכנסה)**

## עקרון מנחה

מסמך חשבונאי שעבר הנפקה (`issued`) **לא ניתן לשינוי**.
התיקון היחיד האפשרי הוא הפקת תעודת זיכוי (`credit_note`) — מסמך חדש.

## מחזור חיי מסמך

```
draft → issued → locked → credited
              ↘
                voided_before_delivery (מ-draft בלבד, לפני הנפקה)
```

### שלב 1: טיוטה (draft)
- ניתן לעריכה חופשית
- מסומן בבירור: "טיוטה — לא לשימוש רשמי" (בדפוס ובמסך)
- לא מקבל `docNumber` ולא `immutableSnapshotHash`

### שלב 2: הנפקה (issueDoc)
- מתבצע על ידי `issueDoc()` ב-`AccountingDocsRepository`
- כל הפעולות בתוך `FirebaseFirestore.runTransaction()` אחד:
  1. בדיקת `status == draft`
  2. קריאת מונה מ-`accounting/_root/counters/{docType}`
  3. הגדלת מונה ב-1 (אטומי)
  4. חישוב `immutableSnapshotHash` (ראה להלן)
  5. עדכון המסמך: `status → issued`, `docNumber`, `issuedAt`, `immutableSnapshotHash`

### שלב 3: לאחר הנפקה
- שדות מוגנים **לא ניתנים לשינוי** (Firestore Security Rules)
- `deleteDoc()` תמיד זורק `UnsupportedError`

## שדות מוגנים לאחר הנפקה

Firestore Security Rules אוכפים שהשדות הבאים לא ישתנו לאחר `status == issued`:

| שדה | הגנה |
|------|------|
| `docNumber` | `unchanged('docNumber')` |
| `issuedAt` | `unchanged('issuedAt')` |
| `immutableSnapshotHash` | `unchanged('immutableSnapshotHash')` |
| `createdAt` | `unchanged('createdAt')` |
| `createdBy` | `unchanged('createdBy')` |
| `companyId` | `unchanged('companyId')` |

## אכיפה — Firestore Security Rules

```javascript
match /accountingDocs/{docId} {
  allow update: if isCompanyMember(companyId)
    && hasAnyRole(['accountant', 'admin', 'super_admin'])
    && unchanged('createdAt') && unchanged('createdBy') && unchanged('companyId')
    // accountant can only edit draft documents
    && (hasAnyRole(['admin', 'super_admin']) || resource.data.status == 'draft')
    // docNumber, issuedAt, immutableSnapshotHash immutable after issued
    && (!('docNumber' in resource.data) || resource.data.status == 'draft' || unchanged('docNumber'))
    && (!('issuedAt' in resource.data) || resource.data.status == 'draft' || unchanged('issuedAt'))
    && (!('immutableSnapshotHash' in resource.data) || resource.data.status == 'draft' || unchanged('immutableSnapshotHash'));
  allow delete: if false;
}
```

> **הערה חשובה**: Firestore Security Rules אוכפים אי-שינוי ברמת שדה בודד, ללא תלות בשיטת הכתיבה (single write / batch write / transaction). כל מסמך בתוך batch נבדק בנפרד מול ה-Rules — אין אפשרות לעקוף.

## חישוב Hash — SnapshotHash.compute()

### שיטה
`SnapshotHash.compute(doc)` — מחלקה `SnapshotHash` ב-`lib/features/owner_dashboard/utils/snapshot_hash.dart`

### שדות הנכללים ב-Hash (5 שדות בלבד)
```dart
static Map<String, dynamic> _buildPayload(AccountingDoc doc) {
  return {
    'docNumber': doc.docNumber,
    'issuedAt': doc.issuedAt?.toIso8601String(),
    'customerId': doc.customerId,
    'lines': doc.lines.map((line) => line.toMap()).toList(),
    'totals': doc.totals.toMap(),
  };
}
```

### אלגוריתם
1. בניית `Map<String, dynamic>` קנוני מ-5 שדות
2. סריאליזציה ל-JSON: `jsonEncode(payload)`
3. קידוד UTF-8: `utf8.encode(jsonString)`
4. SHA-256: `sha256.convert(bytes).toString()`

### דוגמה
```dart
final hash = SnapshotHash.compute(doc);
// → "a3f2c1d4e5b6..." (hex string, 64 תווים)
```

> **הבדל מגרסה קודמת**: הגרסה הנוכחית משתמשת ב-JSON canonical serialization (לא StringBuffer concatenation), ומכסה 5 שדות ליבה בלבד (לא כל השדות המוגנים). זה מבטיח determinism מלא ללא תלות בסדר שדות.

## תעודת זיכוי (Credit Note)

### מתי מותר
- רק ממסמך בסטטוס `issued` או `locked`
- לא ניתן לזכות מסמך בסטטוס `draft`, `credited`, `voided_before_delivery`

### תהליך (createCreditNote transaction)
1. בדיקת סטטוס המסמך המקורי
2. יצירת מסמך `credit_note` חדש (בסטטוס `draft`)
3. עדכון המסמך המקורי: `status → credited`, הוספת `creditNoteId`
4. רישום ביומן ביקורת (`audit` collection)

### שדות חובה ב-credit_note
- `references.originalDocId` — מזהה המסמך המקורי
- `references.originalDocNumber` — מספר המסמך המקורי
- `reason` — סיבת הזיכוי (חובה)

## ביטול לפני מסירה (voidBeforeDelivery)

- רק ממצב `draft` (לפני הנפקה)
- מעבר: `draft → voided_before_delivery`
- נדרשת סיבה (`reason`)
- נרשם ביומן ביקורת

## שרשרת שלמות (Integrity Chain)

- כל מסמך שעבר הנפקה מתווסף לשרשרת
- אוסף: `companies/{companyId}/accounting/_root/integrity_chain/{chainId}`
- append-only — עדכון ומחיקה אסורים
- כל חוליה מכילה: `documentId`, `documentHash`, `prevHash`, `chainHash`, `sequenceNumber`
- פונקציה `verifyChain()` מאמתת את כל השרשרת
