# LogiRoute — единый дизайн-гайд

> **Статус:** аудит существующего UI + бенчмарк 4 конкурентов (Onfleet, OptimoRoute, Routific, Bringg).
> **Источник правды по токенам:** `lib/theme/app_theme.dart` — все «факт»-значения ниже извлечены оттуда.
> Любое значение, которого ещё нет в коде, помечено **(предложение)** — это нормализация, а не действующий токен.
> Визуальные референсы конкурентов и ссылки — в `docs/design-references.md`. Живые дашборды конкурентов не скриншотились; использованы публичные материалы; HEX конкурентов — приблизительные.
>
> **Контекст продукта:** логистика «последней мили». Две принципиально разные среды:
> **водитель на телефоне** (на ходу, перчатки, солнце, RTL/иврит) и **диспетчер/owner на десктопе**
> (плотные данные, карта + таблицы, мышь). Тема одна, фиксированная (по правилу проекта).

## 0. Что уже есть и что несогласованно (итог аудита)

**Хорошо (есть и работает):**
- Единая `AppTheme` с компонентными темами почти на все Material-виджеты (button/input/card/dialog/chip/nav/dataTable…).
- Семантические токены палитры (`accent/green/warning/danger/muted/border/...`) — основа есть.
- Свой компонент вкладок-«пилюль» `LogiRouteTabBar` (+ `PillSelector`, `PillToggleBar`) с hover/press/RTL.
- RTL включён осознанно: `Directionality` на ключевых экранах; `fontFamily: NotoSansHebrew`.
- Роле-ориентированная навигация: driver (mobile-first), dispatcher (tabs), owner (sidebar/drawer).

**Несогласованно (требует унификации):**
1. **Хардкод Material-оттенков** `Colors.orange/green/red/blue/grey.shadeNNN` — массово: только `driver_dashboard.dart` ~45 вхождений, плюс диалоги/баннеры. Это **обходит токены**, ломает консистентность и делает невозможной смену темы.
2. **Две палитры** (`darkPalette` + `lightPalette`) и механизм `setDark/_isDark` живут в коде, **но правило проекта — одна фиксированная тема**. По умолчанию `_isDark = false` → активна **светлая**. Тёмная — «спящий» путь.
3. **Брейкпоинт-зоопарк**: `MediaQuery…width < 600` повторяется в десятках файлов как «магическое число», без общей константы; единственный порог (нет планшетного уровня).
4. **Шкала отступов нестрогая**: вперемешку `2/4/6/8/10/12/14/16/24/32`.
5. **Радиусы**: `4/8/10/12/14/16/18/20/24` — много значений без правила.
6. **Типографика без иерархии**: `textTheme` назначает `w700` почти всем ролям и **без размеров** → экраны задают `fontSize` локально (11/12/13/14/16/18…). Реальной шкалы h1…caption нет.
7. **Статусы фрагментированы**: GPS/маршрут/доставка раскрашены локальными `Colors.*.shade*` (driver), а не едиными токенами статуса.
8. **Таблицы в двух парадигмах**: настоящий `DataTable` (owner/admin/accounting — 6 файлов) и «таблица из карточек» `ListView` (dispatcher/driver) — разная плотность и взаимодействие.
9. **Эмодзи в UI-строках** (`✅ 📍 ⏳ 🚛`) вместо иконок набора.

Дальнейшие секции дают целевые значения; расхождения с кодом отмечены **(предложение)**.

## 1. Принципы дизайна

1. **Плотность данных без шума.** Диспетчер видит десятки маршрутов и сотни точек. Приоритет — информационная плотность (компактные строки, тонкие разделители), но с чётким визуальным ритмом по 8-px сетке. (Заимствовано у Onfleet Table View и Bringg.)
2. **Быстрый скан и «иголка в стоге».** Цвет = статус, не украшение. Один взгляд → понятно, что просрочено/упало/в пути. Семантика статусов едина по всему продукту (см. §5).
3. **Две среды — два режима плотности.** *Driver mobile*: крупные тач-цели (≥44 dp), одна задача в фокусе, минимум текста, действия большим пальцем. *Dispatcher/Owner desktop*: плотные таблицы, карта-первичность, мышь/hover/drag&drop.
4. **Карта — первый класс.** Везде, где есть геоданные, карта равноправна со списком (паттерн Onfleet/Routific/Bringg: unified map+list).
5. **RTL-first, три языка (he/ru/en).** Иврит — основной язык интерфейса; макет обязан зеркалиться корректно (логические отступы, иконки направления, выравнивание).
6. **Один источник стиля.** Любой цвет/отступ/радиус — из токена `AppTheme`, не из `Colors.*`. Новый экран не «изобретает» оформление.

## 2. Сетка и лейаут

### Брейкпоинты (предложение — нормализовать существующий `< 600`)
| Имя | Ширина | Среда | Навигация |
|-----|--------|-------|-----------|
| `xs` | < 600 dp | телефон (driver, dispatcher mobile) | AppBar + bottom/tabs, Drawer; контент в один столбец |
| `sm` | 600–904 | большой телефон/портрет-планшет | как `xs`, но 2 колонки в формах |
| `md` | 905–1239 | планшет/ноутбук | sidebar-rail сворачиваемый |
| `lg` | ≥ 1240 | десктоп (dispatcher/owner) | постоянный sidebar 220 dp + контент |

> **Факт:** в коде сейчас единственный порог `width < 600` (dispatcher `narrow`, owner `isNarrow`). Рекомендация — вынести в `AppBreakpoints` и добавить `lg` для desktop-owner.

### Контейнеры / сетка
- **Базовая сетка — 8 dp** (4 dp для микро-выравнивания). Все отступы кратны 4.
- **Боковая навигация owner:** ширина **220 dp** (факт: `SizedBox(width: 220)`), разделитель `VerticalDivider(width:1)`.
- **Контентная зона:** на формах ограничивать `maxWidth ≈ 420–720` (факт: pick-company `maxWidth: 420`), чтобы строки ввода не растягивались на весь 4K.
- **Карта vs список:** desktop — сплит (карта ≥ 60% по площади, список/таблица сбоку); mobile — вкладки (паттерн dispatcher: «Точки / Маршруты / Карта»).

### Паттерн навигации по ролям
- **Driver (mobile):** `AppBar`(брендовый фон) → стек статус-баннеров (GPS/undo/др. маршрут) → карта (web) или сводка (Android) → `ListView` карточек точек. Активная точка раскрыта, остальные — компактные строки.
- **Dispatcher (mobile+desktop):** `AppBar` + `NotificationBell` + действия → полоса загрузки водителей → `LogiRouteTabBar` (3 пилюли) → `IndexedStack` → `FAB`(ы). На `< 600` вкладки скроллятся.
- **Owner (desktop):** топ-бар (чип компании + бейдж тарифа) → опц. баннер «просмотр как…» → слева сгруппированный список секций (4 группы с подзаголовками и разделителями), справа контент. На `< 600` — `Drawer`.

## 3. Размеры карточек

Базовая карточка — `CardThemeData` (факт): `elevation: 0`, `margin: symmetric(h:12, v:6)`, форма `RoundedRectangle(radius 14, border 1px = c.border)`.

| Тип карточки | Где | Ширина | Высота | Радиус | Фон / бордер |
|--------------|-----|--------|--------|--------|--------------|
| **KPI-плитка** | owner overview, сводки | min 140 / max 240 (предложение) | ~88–120 | 14 | `surface` + `border`; token `tile` для фоновой подложки |
| **Карточка-сущность** (точка/маршрут/счёт) | dispatcher, accounting | заполняет колонку (full-width в списке) | по контенту | 14 | `surface` + `border 1px` |
| **Строка-точка водителя** | driver list | full-width, `margin v:2` (факт) | компактная | **8** (факт) | активная — выделена (см. §5); завершённая — `opacity 0.38` |
| **Баннер/алерт** | статусы GPS, view-as | full-width | по контенту | 0 (полоса) / 10 | semantic-bg + нижний бордер 2px |
| **Диалог-карта** | формы/мастера | `maxWidth 420–560` | по контенту, скролл | **16** (факт) | `surface` |

**Тени:** базово `elevation 0` + 1px бордер (плоский премиум-стиль). Приподнятость — только для выделенных/выбранных элементов: `BoxShadow(color: accent @0.28, blur 8, offset (0,2))` — факт из `LogiRouteTabBar` для выбранной пилюли. **Правило:** тень несёт смысл (выбор/перетаскивание), а не декор.

## 4. Отступы (spacing)

**Единая шкала (предложение, покрывает реальные значения кода):**

| Токен | dp | Применение |
|-------|----|-----------|
| `space-1` | 4 | микро-зазор иконка↔текст, бейджи |
| `space-2` | 8 | базовый зазор внутри строки, между чипами |
| `space-3` | 12 | внутр. паддинг карточек/диалогов, гор. отступ списка |
| `space-4` | 16 | паддинг секций, отступ полей формы, гор. паддинг баннеров |
| `space-5` | 24 | разделение блоков, отступ заголовка секции |
| `space-6` | 32 | крупные пустые состояния, паддинг центр-экранов |

**Правила:**
- Только значения из шкалы. **Изъять из обихода 6/10/14** (предложение): `6→8` (кроме вертикали карточек `v:6`, оставить как пол-шага), `10→8/12`, `14→12/16`.
- Вертикальный ритм карточек: внешний `margin v:6`, внутренний `padding 12` (факт — сохранить).
- Баннеры: `padding symmetric(h:16, v:8…12)` (факт — на шкале).
- Тач-цель driver-кнопок: высота ≥ 32 (факт), **рекомендация поднять до 44** для основных действий на телефоне (предложение).

| `green` (secondary) | `#1FA34A` | успех/доставлено |

`ColorScheme` (факт): `primary=accent, onPrimary=#FFF, secondary=green, surfaceContainerHighest=surfaceHi, onSurfaceVariant=muted, error=danger, outline=border`.

### 5.2 Тёмная палитра (факт, в коде есть, но тема фиксированная → не переключается)

`bg #0A1430 · surface #0E1A38 · surfaceHi #16284E · accent #2F7BE5 · accentSoft #7FB2FF · green #34C759 · greenSoft #6FE6A6 · warning #F5B43C · danger #E96B6B · text #FFFFFF · muted #9FB0CE · border rgba(255,255,255,.12) · tile rgba(255,255,255,.08)`.

> **Предложение (решить с владельцем):** либо (A) официально включить переключатель тёмной темы (палитра уже готова и качественная — «премиум-navy»), либо (B) удалить `darkPalette/setDark/_isDark` как мёртвый код, оставив одну тему по правилу проекта. Сейчас — половинчатое состояние.

### 5.3 Семантические токены статусов (предложение — единый маппинг)

Сейчас статусы раскрашены локально (`Colors.green.shade*`, `Colors.blue.shade*`, `Colors.orange.shade*`). Предлагается **один словарь**, привязанный к токенам темы и согласованный с цвет-кодом Onfleet (grey→purple→blue→green/red):

| Статус (домен) | Токен | Светлый HEX | Семантика |
|----------------|-------|-------------|-----------|
| **Pending / не назначено** | `muted` / нейтральный | `#5A6B8C` (Onfleet: grey) | точка без водителя/маршрута |
| **Assigned / назначено** | `accent` | `#2F7BE5` (Onfleet: purple→у нас синий) | назначено водителю, не начато |
| **In transit / в пути** | `info` (предложение) | `#2F7BE5` светлее / `accentSoft` | едет к точке |
| **Arrived / на месте** | `warning` | `#B97908` | прибыл, ждёт закрытия (таймер) |
| **Completed / доставлено** | `green` | `#1FA34A` (Onfleet: green) | успешно |
| **Failed / отмена / провал** | `danger` | `#C53A3A` (Onfleet: red) | не доставлено/отменено |

GPS-статусы водителя (факт сейчас — `Colors.green/amber/red/orange/grey.shade`) предлагается также свести к этим токенам:
`active→green · waiting→warning · error/disabled/permission→danger · stopped→muted`.

> Для «подложек» баннеров статуса использовать `token.withValues(alpha: 0.10–0.14)` + текст полным токеном (вместо `Colors.x.shade50/900`).

### 5.4 Фон/поверхности (иерархия)
`bg (#F2F5FB) < surface (#FFF) < surfaceHi (#E9EEF8)`. Бордер `border` разделяет поверхности; теней по умолчанию нет.

### 5.5 Приблизительные палитры конкурентов (для бенчмарка; точные брендбуки публично не опубликованы)
- **Onfleet:** светлый дашборд; **статус-цвета карты — факт из демо**: grey/purple/blue/green/red. Бренд-акцент — тёпло-красный/коралл (приблизительно).
- **OptimoRoute:** синий primary + оранжевые акценты; маршруты — мультицвет по водителю (приблизительно).
- **Routific:** **синий + золотисто-оранжевый** (из разбора логотипа), минималистичный светлый UI.
- **Bringg:** фиолетово-синий primary, светлый enterprise-UI (приблизительно).

**Вывод для LogiRoute:** наш `accent #2F7BE5` хорошо ложится в «логистический синий» (Routific/OptimoRoute). Уникализируемся не цветом бренда, а **дисциплиной статусных токенов** (как Onfleet).

## 6. Типографика

**Семейство (факт):** `fontFamily: 'NotoSansHebrew'`, `fontFamilyFallback: ['NotoSans']`. Noto обеспечивает иврит + кириллицу + латиницу → один шрифт на 3 языка. **Сохранить.**

### Шкала (предложение — заменить «всё w700 без размеров» на иерархию)
| Роль | size / line-height | weight | Применение |
|------|--------------------|--------|-----------|
| `display` / h1 | 28 / 34 | 700 | крупные числа KPI, главный заголовок |
| `h2` (headline) | 22 / 28 | 700 | заголовок секции |
| `h3` (title) | 18 / 24 | 700 | заголовок AppBar/диалога (факт: 18/700) |
| `subtitle` | 16 / 22 | 600 | подзаголовок, значение KPI (факт: 16) |
| `body` | 14 / 20 | 600 | основной текст, строки (факт: 14) |
| `body-sm` | 13 / 18 | 600 | вторичный текст, ячейки таблиц, чипы (факт: 13) |
| `label` | 12 / 16 | 700 | метки навигации, бейджи (факт: 12) |
| `caption` | 11 / 14 | 600 | подписи, стат-лейблы (факт: 11) |
| `micro` | 10 / 12 | 700 | мини-бейджи на карточках (факт: 10) |

> **Расхождение:** текущий `textTheme` ставит `w700` почти всем ролям и **без `fontSize`** → экраны задают размер вручную. Предложение — прописать размеры в шкале выше и снизить «болдовость» body до `w600`, заголовки оставить `w700`.

### RTL / иврит
- Весь текст — логическое выравнивание (`start/end`, не `left/right`).
- Иврит читается RTL; UI зеркалится через `Directionality` (факт — на ключевых экранах). **Рекомендация:** включать RTL на уровне `MaterialApp.locale`, а не точечно, чтобы новые экраны наследовали направление.
- Числа/коды/телефоны/`clientNumber` оставлять LTR-островками внутри RTL.
- Noto покрывает огласовки иврита; не подменять системным шрифтом.

## 8. Навигация

### Структуры по ролям
- **Driver:** один экран, без вложенной навигации. `AppBar`(брендовый) + `DriverAppBarActions`. Вертикальный стек: баннеры статуса → карта/сводка → список точек. Переключение «другой маршрут» — через тап-баннер.
- **Dispatcher:** `LogiRouteTabBar` (пилюли) на 3 раздела: **Точки / Маршруты / Карта**. `FAB` контекстный (на вкладке «Точки» — создать клиента/точку).
- **Owner:** сгруппированный sidebar (факт, `_navGroups`): **Обзор** (onboarding, overview) · **Управление** (users_roles, billing, settings) · **Операции** (ops_health, reports, create_document, clients, accounting) · **Соответствие** (audit). На `< 600` → `Drawer` с тем же деревом.

### Состояния пунктов (факт + нормализация)
- **Активный:** `accent`/`primary` цвет иконки и текста, `FontWeight.bold`, `ListTile(selected: true)`.
- **Hover (pill, факт):** фон `surfaceHi` (surface-вариант) или `white@0.28` (on-primary).
- **Press (pill, факт):** `AnimatedScale 0.95`, 90 ms.
- **Выбранная вкладка (факт):** фон `accent`, текст белый, тень `accent@0.28`.
- **Группы owner:** подзаголовок `labelSmall` цветом `primary` + `Divider` между группами (факт).

### Табы / хлебные крошки
- Основной паттерн переключения разделов — **пилюли** `LogiRouteTabBar` (radius 20 внутри контейнера radius 24, бордер `border`) — это фирменный элемент, использовать везде вместо стокового `TabBar` (legacy-`tabBarTheme` помечен как устаревший в коде).
- Хлебные крошки в продукте не используются (глубина навигации мала) — не вводить без необходимости. Контекст даёт заголовок секции + чип компании (owner top-bar).

## 9. Стиль таблиц

Сейчас сосуществуют два подхода — **унифицировать по среде** (предложение):

### A. Десктоп — настоящий `DataTable` (owner/admin/accounting, факт — 6 экранов)
Тема `DataTableThemeData` (факт): заголовок `muted, w700, 13`; ячейка `text, w600, 13`; `dividerThickness 0.6`.
- **Плотность:** компактная; высота строки ~40–44; гор. паддинг ячейки 12–16.
- **Заголовки:** `muted` капс-семантика, **sticky header** при скролле (предложение — как Onfleet/Bringg).
- **Сортировка:** клик по заголовку, стрелка-индикатор (паттерн Onfleet Table / Bringg).
- **Зебра:** по умолчанию **без зебры**; разделение — тонкий `border 0.6`. Если строк много — лёгкая зебра `tile`-подложкой на чётных (предложение).
- **Действия в строке:** иконки-кнопки в конце строки (end в RTL) + контекст-меню `…`; на hover подсвечивать строку `surfaceHi`.
- **Bulk-actions:** чекбокс-колонка + панель массовых действий (optimize/reassign/delete) — заимствование у **Onfleet Table View**.
- **Колонки:** настраиваемый набор/порядок (column-picker + drag) — заимствование у **Bringg** (column-иконка + grip-dots).
- **Пагинация:** для больших наборов — `DataTable` + контролы «N на странице»; для потоков — виртуализация/`ListView.builder`.

### B. Мобайл — «таблица из карточек» (dispatcher/driver, факт)
- `ListView.builder` карточек/строк вместо горизонтального скролла таблицы.
- Плотные строки: бейдж-номер (факт: `CircleAvatar 24`), имя клиента (`14/w700`), адрес (`12/w600 muted`), статус-иконка в конце.
- Завершённые — `opacity 0.38` + strike-through (факт) — сохранить.

### Пустые состояния (предложение — единый компонент)
Иконка `outlined` 48–72 цветом `muted` + заголовок `h3` + пояснение `body muted` + (опц.) primary-кнопка действия. Факт-примеры уже есть (`noActivePoints`, `_NoCompanyScreen`, `_ErrorScreen`) — свести в один виджет `EmptyState`.

## 10. Стиль форм

Базируется на `inputDecorationTheme` (факт).

- **Поля ввода:** `filled: true`, fill = `surfaceHi` (тёмная) / `#FFF` (светлая); бордер `OutlineInputBorder(radius 12, border-color = border)`; **focus** — `accent, width 1.6`; **error** — `danger`. (Всё факт.)
- **Лейблы:** `labelStyle` цветом `muted, w600`; hint `muted, w500`. Лейбл над/в рамке поля; в RTL — выравнивание `end`.
- **Helper / ошибки:** текст ошибки `danger`, 12–13; helper `muted`. Валидация — по `errorBorder` + текст под полем (не только цвет — для доступности).
- **Кнопки — иерархия (факт-темы):**
  - **Primary** — `FilledButton`/`ElevatedButton`: фон `accent`, текст белый, `radius 12`, `w700`, `elevation 0`.
  - **Secondary** — `OutlinedButton`: текст `accentSoft`, бордер `border`, `radius 12`.
  - **Tertiary** — `TextButton`: текст `accentSoft`, без фона.
  - **Деструктив (предложение):** primary-кнопка с фоном `danger` (сейчас местами `Colors.red.shade700` / `Colors.orange` — заменить на токен).
  - **Один primary на диалог/форму.** Порядок в RTL: primary — в логическом начале строки действий.
- **Диалоги:** `DialogThemeData` (факт): фон `surface`, `radius 16`, заголовок `18/w700`, контент `14/w600 muted`. Контент-паддинг 12–16. Ширина — `maxWidth 420–560`.
- **Bottom sheets (факт):** `radius top 18`, фон `surface` — для мобильных действий (driver POD и т.п.).
- **Мастера (wizard):** многошаговые формы (company setup, import mapping — факт, есть `…wizard_screen.dart`). Паттерн: степпер сверху/сбоку + один primary «Далее» + tertiary «Назад»; прогресс-бар `accent` на `surfaceHi`-треке (факт `progressIndicatorTheme`). Заимствование у Onfleet/OptimoRoute «Planning Settings» поп-апов: настройки оптимизации — модалкой-мастером, а не разрозненными полями.
- **Переключатели/чекбоксы/радио (факт):** selected = `accent`, иначе `muted`. Switch trackColor selected = `accent`.
- **RTL форм:** порядок полей, иконки prefix/suffix и кнопки действий зеркалятся; числовые поля (координаты, суммы, `clientNumber`) — LTR-островки.

## Что заимствуем для LogiRoute и почему

1. **Статус-цвета как у Onfleet (grey→assigned→in-transit→green/red).**
   *Почему:* единственный твёрдый публичный факт по цвету; решает нашу проблему №7 (фрагментация статусов). → §5.3, единый словарь токенов вместо `Colors.*.shade`.
2. **Desktop-таблица в духе Onfleet Table View + Bringg.**
   *Почему:* диспетчер/owner работают с большими объёмами. Берём: sort по заголовку, sticky header, **bulk-actions** (Onfleet), **column-picker с grip-dots** (Bringg). → §9A.
3. **Карта-первичность + drag&drop (Routific/OptimoRoute).**
   *Почему:* у нас уже есть drag-assign на карте и reorder маршрута — узакониваем как ядро UX, а не побочную фичу. → §2, §1.4.
4. **Timeline-плитки заказов (OptimoRoute) — опционально для dispatcher.**
   *Почему:* быстрый визуальный баланс загрузки водителей по времени; у нас уже есть `DriverWorkloadPanel` — развить до тайм-лайна. (предложение)
5. **Мастер настроек оптимизации (OptimoRoute «Planning Settings»).**
   *Почему:* собрать параметры маршрута (выезд/скорость/разгрузка — они уже в `CompanySettings`) в один понятный поп-ап-мастер. → §10.
6. **Минимализм форм Routific (drag вместо полей).**
   *Почему:* меньше строк/полей = меньше ошибок водителя/диспетчера; согласуется с принципом «плотность без шума». → §1, §10.

**Что НЕ копируем:** тёмные «командные» дашборды и сложные BI-конструкторы (Bringg enterprise) — избыточно для нашего масштаба; остаёмся на одной светлой теме (правило проекта) и фиксированном наборе секций.
